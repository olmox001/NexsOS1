# Technical Report: AMD64 Hybrid Boot Stabilization
**Project**: OS1 Microkernel
**Date**: May 13, 2026
**Subject**: Stabilization of Hybrid ISO Boot and Userland Mounting

## 1. Executive Summary
This report documents the resolution of critical boot-time failures for the AMD64 architecture. The primary objective was to ensure stable loading of the user-space environment from a Hybrid ISO (BIOS/EFI). We achieved full stability by refactoring the bootloader-to-kernel transition, implementing a multi-format partition parser, and optimizing VirtIO-Block device interaction.

## 2. Technical Challenges & Solutions

### 2.1. Multiboot Magic Preservation
- **Problem**: RDMSR/WRMSR operations during the 32-to-64-bit transition were clobbering the `%eax` and `%ebx` registers, losing the Multiboot magic number (`0x36d76289`) and the information pointer.
- **Solution**: Refactored `kernel/arch/amd64/boot/start.S` to save these registers on the boot stack immediately upon entry and restore them precisely before the final jump to `kernel_main` in 64-bit mode.

### 2.2. Hybrid ISO Partition Mapping
- **Problem**: `grub-mkrescue` produces a Hybrid ISO with an MBR partition table. The kernel's partition parser only supported GPT, failing to find the userland Ext4 partition appended by `xorriso`.
- **Solution**: 
    - Implemented a legacy **MBR Parser** as a fallback in `kernel/fs/gpt.c`.
    - Modified the `Makefile` to extract the raw Ext4 partition (`userland.img`) using `dd` and appending it as a direct partition (MBR Index 2).
    - This allows the kernel to mount the filesystem without needing to handle nested partition tables.

### 2.3. VirtIO-Block Driver Optimization
- **Problem**: The system would hang during sector reads when using Legacy VirtIO interfaces on certain ISO images. Additionally, the driver used `static` request structures, which were not safe for the multi-core environment (4 CPUs).
- **Solution**:
    - Forced **Modern VirtIO** (v1.0+) in QEMU release flags (`disable-legacy=on`).
    - Refactored `virtio_blk_read/write` to use stack-allocated request structures, ensuring multi-CPU safety.
    - Standardized timeouts and improved interrupt acknowledgment logic.

## 3. Infrastructure & Git Management
- **Issue**: The generated Hybrid ISO exceeded 100MB (119MB), triggering GitHub's file size limit and blocking repository pushes.
- **Resolution**:
    - Reduced the default disk image size in `tools/mkdisk.c` from 128MB to **96MB**.
    - This resulted in an 87MB ISO, fitting safely within GitHub's limits.
    - Cleaned the Git history by resetting the local branch and re-committing code changes without large binary artifacts.
