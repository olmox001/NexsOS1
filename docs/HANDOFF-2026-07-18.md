## 3. ARCHITECTURE DECISIONS TAKEN (and WHY — do not re-litigate blindly)

**Ports are capabilities (`OBJ_TYPE_PORT`, `OS1_NS_PORT`).** A client addresses a
SERVICE BY NAME, never a pid — the seL4 rule ASTRA §6.5 states and the ambient
`SYS_SEND` violates. Acquiring with READ *publishes* and claims the name (a
second receiver is refused, so a service identity cannot be stolen by racing);
without READ you get a send-only capability. **The sender identity is stamped by
the kernel and is unforgeable** — device-verified, and it is what the service
uses to authorise requests and assign job ownership.

**`SYS_PORT_SEND_CAPS` — rights travel in the message.** A handle index means
something only inside ONE process's table. `cap_grant` would need the service's
PID, dragging pid addressing back into the one path built to remove it. So the
kernel installs transferred handles in the receiver and rewrites the leading
payload slots with the indices IT will see.

**`owner_pid` (logical parent) + `OBJ_CTL_SETOWNER`.** `parent_pid` = who
mechanically spawned; `owner_pid` = who it BELONGS to. They diverge once a
service spawns on a client's behalf. Kill/stop/cont authority is an ancestry
walk, so without this a shell migrated onto the daemon would silently lose job
control over its own jobs. Privileged-only: it DELEGATES authority.

**The service is started by init, never by clients.** `process_create_caps`
applies a monotonic creator clamp, so a PLVL_USER client spawning the service
would get a PLVL_USER service — and the daemon's two defining powers
(`SETOWNER`, taking a client's fds) are BOTH privileged-only. Such a service
answers requests but silently fails to delegate, surfacing as a broken shell.
ASTRA §6.4 already prescribes this ("SRL services are supervised ELF processes").

**No pid key in the registry for the exec service.** The PORT NAME is the
endpoint; it cannot go stale across a respawn, and the new instance republishes
by taking the receive right, so discovery heals itself.

**Request transport = port rendezvous + out-of-line body over a delegated pipe.**
`ipc_message.payload` is 64 bytes; a realistic request exceeds it (a fixed record
measured 1272 B). Truncating argv would corrupt commands silently.

**Libraries phase ≠ services phase** (maintainer). An earlier revision said to do
them together; wrong — different boundaries, independently verifiable.

**`include/abi/` is CONCEPTUALLY WRONG** (maintainer). A shared header directory
renames the kernel↔userland coupling instead of removing it. Target: the kernel
EXPOSES information dynamically as OBJECTS; userland libraries share no logic;
kernel includes live only in the kernel tree. `docs/R0-CENSUS-LIB-SPLIT.md` is
marked SUPERSEDED IN PART at its head for proposing `include/abi/`.

**Phase 9 (b) is TRANSITIONAL, (a)/9d is FINAL.** 9c is staging so the logic can
be finalised against a working system. Phase 9 is not done until interactive
jobs also execute through the service.

## 4. NEXT WORK, IN THE MAINTAINER'S ORDER

### 4.0 — UNIFY THE TWO PER-PROCESS REPRESENTATIONS (decide before 4.1)
Maintainer, 2026-07-18: "abbiamo già una rappresentazione registry per object e
una fd table separata, va tutto supervisionato e unificato secondo astra".

VERIFIED state (measured, not assumed):
- the kernel fd table IS already absorbed into the handle table
  (`kernel/include/kernel/object.h`: "a POSIX descriptor IS a handle, fd N ==
  handle N") — there is NO parallel fd array, kernel or libc;
- `regfs` already exposes the registry as files under `/reg`;
- **the real duplication**: per-process state exists TWICE — the kernel's
  authoritative HANDLE TABLE + capability state, and the `sys.proc.<pid>.*`
  registry keys, which are written by USERLAND (nxexec) and are best-effort.
  The proof that this is a defect and not a design: those keys can outlive their
  process, which is why `nxexec_prune_identities()` had to be written to garbage
  collect them.

Unification per Phase 5's already-chosen VIRTUALISED form: the registry view of
a process must be BACKED by kernel state, not a userland copy.  That deletes the
staleness class and the pruner outright, rather than maintaining them.

**This gates 4.1**: putting environment in userland-written
`sys.proc.<pid>.env.*` keys would ADD a second stale copy — the exact thing
being removed.  Environment should live in the kernel's per-process state and be
exposed through the same virtualised namespace.

Already landed toward this: `proc_get_lineage()` + `parent=`/`owner=` in the
PROCESS object's status block, so inheritance can follow the LOGICAL owner
(required since 9c: the mechanical parent is now the service, so inheriting via
parent_pid would inherit from nxexec instead of from the requester).

### 4.1 — `env` + environment variables (after 4.0's decision)
The lua suite stops at `main.lua:133`: `RUN('env LUA_INIT= LUA_PATH=x lua %s > %s')`.
There is no `/bin/env`, and `getenv()` is a **static table** (`PATH`, `HOME`,
`TMPDIR`, `SHELL` — `lib.c` ~1786). Two pieces: an `env` utility, and real
environment propagation to children (spawn carries no environment today). Note
this interacts with the exec service: the request would need to carry env, which
the variable-size body can already express.

### 4.2 — Phase 16: ROADMAP §1.C blockers (gates 9d)

## 5. TRAPS — things that will bite a fresh context

- **`&&` silently does half the line.** `cd X && cmd` performs only the `cd`.
  nxshell has no sequencing (`&&`, `||`, `;`). Run commands separately.
- **QEMU holds the disk image lock.** `make run` fails with "Failed to get write
  lock" if an instance is open. Guard: `pgrep -f qemu-system-aarch64`.
- **Force-included headers must be Makefile prerequisites.** `lua_portability.h`
  wasn't, so editing it left STALE objects and a fix appeared not to work.
  Proven with `nm -u lua.o`. Same class can recur for any `-include`.
- **`struct spawn_redir` grew a `source_pid` field.** EVERY constructor must set
  it; leftover garbage selects the privileged take-from-another-process path.
- **Two `elf.h` files share the guard `_KERNEL_ELF_H`** and the kernel one never
  DEFINES it. If the userland header is included first, the kernel header's body
  is silently skipped — which ELF definitions exist depends on include ORDER.
  Latent, invisible in a passing build. Not yet fixed.
- **The intermittent SMP boot race** (~1 in 4 boots) predates this work; it is in
  the K3-userland bring-up window (ASTRA §7.9-7.10), not caused by these changes.
- **Economical review agents hallucinate.** Both rounds produced confident false
  positives — including one agent quoting MY OWN text back as another document's
  contradiction. Verify every claim against the code before acting on it.

## 6. VERIFICATION — the discipline that has been working

`captest` (`user/bin/captest.c`) is the system regression suite, organised by
LAYER so a failure says which stratum broke: capability/object → ports → process
ownership → exit-status retention → pipes → POSIX surface → exec service.
**Currently 47/47 PASS on device.** Extend it every phase — the maintainer asked
for exactly this.

Writing the test BEFORE/WHILE implementing has caught **seven** defects that a
clean build did not: port unpublish race, `src` shadow collision, pipes
undelegatable (missing `TRANSFER`), handle indices read from the wrong field,
leaked probe pipe, shadowed `rpid`, and the `SYS_WAIT` status hole.

```sh
make                      # build (aarch64)
make run                  # boot QEMU (close any open instance first)
# on device:
captest                   # expect 47/47
cd /home/LUA/luatest      # then, separately (no &&):
lua all.lua               # expect: stops at main.lua:133 (missing `env`)
```

The lua suite is the best regression signal we have for the execution path —
`os.execute` is exercised constantly. **Unchanged stopping position after a
refactor is the CORRECT result**; an unexpected advance deserves suspicion.
