# DESIGN 2026-07-23 — On-disk persistence, partition model, first-boot install

> **Phase F1.**  Required by the project's own R6 rule ("doc dedicato prima del
> codice"): this changes the boot contract, the disk format and the write path
> at once, so the design is agreed before F2 writes any code.
>
> Method: **ASTRA** — every device and format is a provider behind a contract;
> `arch/` holds only ISA/boot glue; the kernel core never learns "which kind of
> disk".  Extends `MICROSCOPE-RELEASE-STORAGE.md` (R1 block contract landed;
> R2 tmpfs / R3 on-disk persistence / R4 memory drivers open), not a new track.

## 1. What already exists (verified in source, not assumed)

Read out of the tree before designing anything:

| mechanism | state | evidence |
|---|---|---|
| Block layer as a CONTRACT with one active backend | **exists** | `kernel/drivers/block/block.c`: `block_register()`, `block_read/write()` → `static const struct block_dev *active` |
| virtio-blk provider; ramdisk provider OVERRIDES it when a boot module is present | **exists** | `kernel/main.c:236-240` `virtio_blk_init(); ramdisk_init();` |
| ramdisk writes are volatile BY CONSTRUCTION | **exists** | `ramdisk.c`: `memcpy(disk, buf, len); /* RAM-backed: writes are volatile */` |
| ext4 is **WRITE-THROUGH**: every write reaches the device before the call returns | **exists** | `ext4_bwrite()` → `block_write()`; `ext4.c:82` `NOTE(EXT4-15): bypasses the buffer cache` |
| The buffer cache is **DEAD CODE** | **defect** | zero callers of `buffer_get`/`buffer_put` outside `buffer.c`; `buffer_sync()` is invoked only from `buffer.c:191`, its own eviction path.  It has never been in the I/O path |
| **There is no persistence ARCHITECTURE** (maintainer, confirmed) | **gap** | a write lands in `disk.img` on `make run`, but `disk.img` is a BUILD ARTEFACT regenerated by `make`; the shipped release boots the volatile ramdisk; there is no installed state, no user data that survives an image rebuild, and no crash-consistency |
| Partition table parsed, with per-partition **type GUID** already carried | **exists, unused** | `kernel/include/kernel/gpt.h`: `struct partition { start_lba, end_lba, size_sectors, index, type_guid[16] }` |
| Root mount selection = **first partition any FS driver accepts** | **exists** | `vfs.c vfs_init()`: `for i in 0..num_partitions { for d in drivers { if mount(...)==0 → mounts[0] = "/" } }` |
| Non-root mounts + unmount with in-flight pinning | **exists, unused** | `vfs_mount_at(mountpoint, ops, fs_private)`, `vfs_umount()` (-EBUSY while pinned) |
| Buffer cache is **write-back** with dirty flags | **exists** | `buffer.h`: dirty flag "set by filesystem code; cleared by buffer_sync()"; `buffer_sync()` writes dirty buffers |
| RAM discovery is an arch-HAL contract | **exists** | `arch_platform_get_mem_regions()` — aarch64 FDT/manual probe, amd64 MB2 MMAP |
| `mkdisk` emits **one** partition, hand-rolled single-group ext4 | **limit** | `tools/mkdisk.c`: `MIN_PARTITION_BLOCKS` 432 MiB, ~1014-inode cap |
| **The KERNEL partition is MISSING** (maintainer, confirmed) | **gap** | `mkdisk.c:835` writes exactly one entry, `e[0].type_guid = TYPE_DATA`.  `TYPE_BOOT` and `TYPE_KERNEL` (lines 60–71) are **orphaned constants** — defined, never used.  MICROSCOPE R1 dropped the boot+kernel partitions ("never read back") when the release moved to the GRUB module path |
| **`TYPE_KERNEL` and `TYPE_DATA` are BYTE-IDENTICAL** | **defect** | both are `0FC63DAF-8483-4772-8E79-3D69D8477DE4` — the standard **Linux filesystem data** GUID.  They cannot distinguish a role, and they claim to be Linux partitions |
| No SHA-256 anywhere; CRC32 exists (GPT header only) | **gap** | `kernel/lib/crc32.c`; no digest over the image |
| Storage drivers | **virtio-blk + ramdisk ONLY** | no AHCI/NVMe/SATA/SCSI anywhere in `kernel/` |

### 1.1 Two landmines this design must defuse (both pre-existing)

- **GPT-02** (documented in `gpt.c`): the GPT path fills `partitions[]`
  0-indexed, the MBR path 1-indexed, and the header warns *"changing the disk
  image layout will silently mount the wrong partition"*.  F2 changes the disk
  layout by definition, so this is directly in the blast radius.
- **Root selection is "first mountable wins"**, not role-aware.  The moment
  there are several ext4 partitions, whichever is probed first becomes `/`.
  Adding partitions without fixing this is a silent mis-mount.

**Decision D1: identify partitions by ROLE (type GUID), never by index.**
`struct partition.type_guid` already exists and is already populated by the GPT
parser — the field is there and unused.  The kernel gets a
`partition_find_by_role(role)` seam; `vfs_init()` mounts the partition whose
GUID says ROOT instead of the first thing that mounts.  This retires GPT-02's
index ambiguity rather than working around it, and it is what makes an
installer able to re-partition without the kernel guessing.

**Decision D10: mint DISTINCT NEXS role GUIDs.**  Not a preference — a
correction.  `TYPE_KERNEL` and `TYPE_DATA` are currently the *same* GUID, so
role identification is impossible with today's constants; and that GUID is
`0FC63DAF-…`, the **Linux filesystem data** type, so another OS's installer is
entitled to treat our partitions as its own.  F2 assigns one distinct,
NEXS-owned GUID per role (KERNEL, ROOT, MACHINE, USRn) and records them here.
`TYPE_BOOT` (`21686148-…`, the BIOS-boot GUID) stays as-is where a BIOS boot
partition is genuinely required — that one is standard on purpose.

## 2. Partition model

```
DISK
 ├── P0  NEXS-META tiny: Merkle root hashes (§4bis), install marker,
 │                 min_accepted_version (D18).  Integrity-protected, readable.
 ├── PK  KEYSTORE  secrets only (password hashes; later the device identity
 │                 private key).  NOT in the VFS namespace, kernel-only
 │                 reader, out of the normal partition view (D20).
 ├── P1  KERNEL    boot chain, TWO SLOTS A/B (D14).  IMMUTABLE at runtime
 │                 except through the authorised update path.
 │                 CURRENTLY ABSENT — mkdisk writes only the DATA partition;
 │                 F2 re-introduces it (see §1: TYPE_KERNEL is an orphan).
 ├── P2  ROOT "/"  the system tree.  Writable by ROOT only.
 ├── P3  MACHINE   machine-only state, so "/" can be root-writable without
 │                 exposing machine-level state to root.
 └── P4  USR1      per-user tree, mounted at /mnt/usr1.  More users → P5, P6…
```

**Role GUIDs.**  Custom NEXS GUIDs (not the EFI/Linux well-known ones): the
roles are ours and reusing e.g. the Linux-root GUID would make another OS's
installer think it owns the partition.  Concrete values are assigned in F2 and
recorded here; the parser matches on them, so nothing else in the kernel needs
to know the numbers.

**Which tree lives where** (this fixes the current all-in-one-partition layout):

| tree | partition | writable by |
|---|---|---|
| Merkle roots, install marker, version floor | P0 NEXS-META | installer + the journal transaction (D6') |
| secrets (password hashes, later device key) | PK KEYSTORE | kernel only; never in the namespace |
| bootloader, kernel image (slots A/B) | P1 KERNEL | only the authorised update path (D14) |
| `/sys`, `/bin`, `/etc`, `/fonts`, `/lib` | P2 ROOT | root (machine for `/sys/bin`) |
| `/system` | P3 MACHINE | machine only |
| `/mnt/usr1/home` (the user's home) | P4 USR1 | that user |

This is what makes the level model REAL rather than path-string-deep: today
`vfs_write_allowed()` protects `/sys/bin` and `/system` by comparing strings
(B2.2).  After F3 the ACL and the PARTITION agree — a machine-only path is also
physically on the machine partition, so a bug in the string ACL cannot expose
it.  **Decision D2: the tree ACL and the partition role must agree; the ACL
consults the role, not just the path.**

## 3. Boot and the RAM copy

```
 P1 KERNEL ──► kernel starts
      │
      ├─ arch_platform_get_mem_regions()  → total usable RAM
      ├─ reserve ~20% of RAM as the "/" working set        (maintainer's figure)
      ├─ mount the system image into it                    (ROOT, from P2)
      └─ nxinit from /sys/bin                              (unchanged)

 kernel + root run from RAM COPIES; the DISK copies stay authoritative.
```

**Decision D3 (REVISED after verification — the first version was wrong).**
The first draft said "the copy IS the buffer cache".  That was built on an
assumption that did not survive reading the code: **the buffer cache has no
users at all**, and ext4 explicitly bypasses it (`EXT4-15`).  Designing the
persistence path on it would mean resurrecting an unused subsystem, carrying its
two known defects (MM-BUF-03 no per-buffer lock, MM-BUF-04 flushes at most 64),
and putting a cache in front of a filesystem that is currently correct precisely
*because* it is write-through.

**What is actually true today:** every `write()` is already synchronous to the
device.  So "persist at the syscall boundary" is not a policy we need to impose —
it is the existing behaviour.  What is missing is not flushing.  It is:

1. **an installed system** (today `disk.img` is a build artefact; the release
   boots a RAM disk),
2. **crash-consistency** (a torn write leaves a corrupt block; there is no
   journal, no atomic replace),
3. **performance** (unbuffered read-modify-write of a 4 KiB block per write),
4. **verifiability under mutation** (§4bis).

**Decision D3': the RAM budget is a READ cache, and the write path stays
write-through until it is made crash-consistent.**  Adding a write-back cache
BEFORE there is a journal converts "a torn block" into "a torn block plus an
unknown amount of lost work" — strictly worse.  Order matters: (a) make writes
atomic, (b) then buffer them.  The "~20% of RAM" is therefore a tunable read
cache budget (Q2: tunable default, per the maintainer), which is also why it
needs no new mapping on either arch — see D4.

**Decision D4 (amd64 asymmetry — the maintainer's flagged caution).**  On amd64
the kernel shares the address space with the current process and identity-maps
usable RAM (`arch/amd64/mm/mmu.c:116`); aarch64 splits TTBR0/TTBR1 and does not.
Sizing a RAM budget at runtime therefore interacts with the identity map on one
arch and not the other.  Because D3 makes the budget a CACHE budget rather than
a mapped RAM disk, this reduces to "how many pages may the buffer cache hold",
which is arch-neutral and needs no new mapping on either side.  That is the
main reason D3 is preferred over a bulk RAM image.  The residual arch work is
stated as a contract in `hal_uaccess.h`'s sibling (HAL-0), not open-coded.

## 4. The persistence contract (tied to the syscalls)

The maintainer's requirement: *"when userland calls write into the VFS on an
authorised disk-saved path, at the end of the operation the file (section) is
passed to the disk driver which handles it."*

```
 userland write(2) ─► syscall ─► VFS ─► vfs_write_allowed()  (level+role ACL)
                                    │
                                    ├─ FS provider writes into the buffer cache
                                    │       (RAM view is now current)
                                    └─ at the END of the operation:
                                          persist(range) ─► buffer_sync(range)
                                                        ─► block_write ─► driver
```

**Decision D5: granularity is the BLOCK RANGE the operation touched.**  ext4
already computes exactly this (`phys_block`, the 8-sector run it read-modify-
writes).  "The file" would re-derive a set the layer below already has and would
rewrite untouched blocks.

**Decision D6 (REVISED — the maintainer's "F4 must not force anything").**  The
first draft decided "persistence is SYNCHRONOUS at the syscall boundary".  That
was imposing a policy on a system that already behaves that way — ext4 is
write-through, so the syscall already returns only after `block_write`.  There
is nothing to force.  **The real gap is ATOMICITY, and that is what F4 must
solve.**

Today a `write()` that spans a block does read-modify-write and issues a single
`block_write`.  If power is lost mid-write, that block is torn: neither the old
nor the new content, with no record that it happened.  No amount of flush policy
fixes this — it is a property of overwriting data in place.

**The correct long-term method is to stop overwriting in place.**  Two
established approaches, both compatible with the existing block contract:

| approach | how it works | cost | fit here |
|---|---|---|---|
| **Journal** (ext4-style) | write the change to a log, then apply it; replay the log at mount | double write for metadata; a log region per FS | matches the on-disk ext4 we already carry |
| **Copy-on-write** | never overwrite a live block; write elsewhere and atomically swing a pointer | needs allocator + reference counting | cleaner invariants, larger change |

**Decision D6': journal the write, then apply it — and make the journal the same
mechanism the Merkle update rides on (§4bis).**  A single atomic unit then
covers "the data changed AND its hash changed", which is exactly the pairing
that must never be observed half-done.  Ordering per file follows syscall order;
a crash replays the journal at next mount and the digest tree is consistent by
construction.

This is deliberately NOT scheduled as one step: F4 splits into **F4a journal +
replay** (atomicity), then **F4b read cache** (performance, safe only once
writes are atomic — D3'), then **F4c Merkle update in the same transaction**.

**Failure semantics.**  A driver-level failure propagates as the syscall's errno
and the journal entry is not committed, so the on-disk state is the pre-write
state — not a partial one.  Failures are reported through the diagnostic channel
(Programme C).  A write the ACL refuses never enters the journal.

**What is NOT persisted:** P1 (kernel) is never writable at runtime; tmpfs
trees (`/tmp`, logs — MICROSCOPE R2) are RAM-only by design; the RAM-only setup
mode persists nothing.

## 4bis. Image validity — SHA-256 over the image

> Maintainer: *"disk.img should be SHA-256 'signed' so the kernel knows it is a
> valid image."*

**Be precise about what a digest buys, because the two are often conflated:**

| property | what it means | what gives it |
|---|---|---|
| **Integrity** | the image is not CORRUPTED (bad flash, truncated copy, half-written install) | a SHA-256 digest |
| **Authenticity** | the image is the one WE built (not tampered with by someone) | a signature over that digest with a key the kernel trusts |

A bare SHA-256 stored next to the image gives integrity only: anyone who can
rewrite the image can rewrite the digest beside it.  That is still genuinely
worth having — it is exactly what catches the failure modes this system will
actually hit first (an interrupted install, a corrupt USB stick, a partially
flashed disk).  Authenticity is a strict superset and needs a root of trust.

**Decision D11: ship the SHA-256 digest now; design the slot so a signature can
occupy it later.**  The metadata record carries `{ digest, algorithm, length,
signature[] }` with the signature field empty in v1.  Adding real signing later
is then a value change plus a verifier, not a format change.

**What is covered.**  The maintainer's decision: **verify ROOT at every boot
too — the seconds do not matter.**  Accepted, but it exposes a contradiction that
a single flat digest cannot survive, and the fix is the thing that makes the
whole feature correct rather than ceremonial:

> **A monolithic SHA-256 over P2 goes stale the instant the system writes to
> P2** — and F4 exists precisely to make P2 writable.  After the first write the
> boot check fails on a healthy system, and the only ways out are to recompute
> the whole digest on every write (absurd) or at shutdown (lost on power cut).
> A check that must be disabled to use the system protects nothing.

**Decision D13: a MERKLE TREE over blocks, not a flat digest.**  This is the
dm-verity / fs-verity design and it is the correct long-term answer:

- each 4 KiB block has a SHA-256 leaf hash; hashes are hashed pairwise up to a
  single **root hash** stored in P0;
- **boot verification** = recompute the tree over P2 and compare the root hash —
  the full-partition read the maintainer asked for, with the same cost;
- **a write updates only the leaf and the path to the root** — `log2(n)` hashes,
  not the partition.  So the tree stays valid under mutation, which is what
  makes "verify at every boot" survive contact with a writable filesystem;
- **reads can be verified continuously and cheaply** (verify the block against
  its leaf), so corruption is caught when it is read, not only at boot.

The Merkle update is committed in the SAME journal transaction as the data
(D6'), so "data changed" and "hash changed" are never observed half-done.

| region | protection | verified |
|---|---|---|
| **P0 NEXS-META** | self-describing; holds the root hashes + marker + version | read first, integrity-checked by its own CRC |
| **P1 KERNEL** | Merkle root (immutable content) | **every boot**, before executing |
| **P2 ROOT** | Merkle root, updated per write (D6' transaction) | **every boot** (maintainer) + per-block on read |
| **P3 MACHINE** | Merkle root, updated per write | every boot (small; holds `/system` + the kernel image per Q3) |
| **P4 USRn** | per-block leaves, root updated per write | on read; full sweep on demand — user data is large and mutable, and a boot-time full sweep of every user partition scales with users, not with the system |

**Decision D12: the metadata record lives in its own GPT partition (`NEXS-META`),
not inside a region it describes.**  A digest cannot live in the bytes it
covers, and stuffing it into GPT entry padding would abuse a table another
tool may rewrite.  A tiny dedicated partition is self-describing, findable by
role GUID (D1/D10), and gives the installer somewhere to record the install
marker (D8) and the image version alongside.

**Where the code goes.**  SHA-256 does not exist in the tree (only
`kernel/lib/crc32.c`, used for the GPT header).  It is needed on both sides:
`tools/mkdisk.c` computes it at build time; the kernel verifies it at boot.
Per ASTRA it is a library primitive (`kernel/lib/sha256.c`) consumed by the
block/partition layer — not something the FS or the driver open-codes.  A
public-domain/MIT single-file implementation is appropriate; it must be the
same implementation on both sides so a mismatch is a real mismatch.

**Failure policy.**  A P1 digest mismatch is a boot-stopping condition reported
through the diagnostic channel (Programme C) — booting code that fails its own
integrity check is precisely what the check exists to prevent.  A P2 mismatch at
install time aborts the install (no marker written, D8), so the next boot
retries rather than leaving a half-verified system.

## 5. Install / first-boot state machine

```
 boot ─► is the install marker present on P2?
          │
          ├─ YES ─► normal boot: mount P2 as "/", P3, P4; shell as the user
          │
          └─ NO  ─► FIRST BOOT: nxinit starts nxsetup
                     │
                     ├─ mode? ── FULL (a disk exists)
                     │            ├─ choose disk           (nxdisk: enumerate)
                     │            ├─ choose partition sizes (P1/P2/P3/P4)
                     │            ├─ nxdisk: partition + format
                     │            ├─ nxcomp: expand the compressed `usr` tree
                     │            ├─ copy the boot chain onto P1  (self-booting)
                     │            ├─ choose username → create P4, seed its env
                     │            └─ write the install marker
                     │
                     └─ RAM-ONLY (test): home in RAM from detected free space,
                        nothing written to disk, marker NOT written
```

**Decision D7: the installer is USERLAND, the privileged work is a SERVICE.**
`nxsetup` is an ordinary app; it asks `nxdisk` (partition/format/mount) and
`nxcomp` (decompress) over ports (`OS1nx_disk`, `OS1nx_comp`, following the
existing `OS1nx_<service>` standard).  Reason: partitioning is the single most
destructive operation in the system, so it belongs in one auditable supervised
service with an explicit capability, not scattered in an app — the same
argument that put execution behind `nxexec`.

**Decision D8: the marker is the authority on "installed", and it lives on P0.**
Not a registry key (the registry is rebuilt from the image) and not a kernel
flag (lost on reboot).  It sits in `NEXS-META` next to the digests (§4bis, D12),
because "is this installed" and "is this image valid" are the same question
asked at the same moment in the boot path.  `nxsetup` runs exactly once because
the marker is the thing that makes it not run.

**Idempotence/safety:** every destructive step is confirmed against the chosen
disk's identity; an interrupted install leaves no marker, so the next boot
restarts the installer rather than booting a half-installed system.

## 6. Compression format

Needed to ship `usr` compressed inside `disk.img` and expand it at install
(F6/F10).  Requirements: GPLv2-compatible, small decoder, no allocator heroics
(it runs early, in a service, on a freestanding libc), and we only need
**decompression** at install time — compression happens on the host in `make`.

| candidate | license | decoder size | notes |
|---|---|---|---|
| **miniz** | MIT | single file, ~2k lines | DEFLATE/zip, self-contained, allocator-light |
| zlib | zlib licence | small | the reference DEFLATE, widely audited |
| LZ4 | BSD-2 | tiny | fastest, weakest ratio; block format is trivial |
| zstd | BSD-3 | large | best ratio, biggest decoder |

**Decision D9 (proposed): miniz, zip container.**  Single-file MIT decoder is
the least intrusive thing to vendor into a freestanding service, DEFLATE ratio
is adequate for a userland tree, and `zip` gives a per-file container so the
installer can expand selectively rather than streaming one blob.  Host side uses
standard `zip`, so `make` needs no new tooling.
*(This is the one decision most worth the maintainer overriding if there is a
preference — the rest of the design does not depend on which codec wins.)*

## 7. Ordering, and what each phase may assume

```
F1 (this doc)
 └─ F2 mkdisk multi-partition + role GUIDs
     └─ F3 kernel: role-aware mount (fixes GPT-02 + "first mountable wins"),
        │        ACL agrees with the partition role (D2)
        └─ F4 persistence write-back (D5/D6) + MM-BUF-03/04 fixes
            ├─ F5 nxdisk service
            ├─ F6 nxcomp service
            └─ F7 nxsetup ─► F8 nxauth (su) ─► F9 default user usr1
                              └─ F10 make integration (compress usr, boot chain
                                     in disk.img, re-check release/tmpfs)
```

F2–F4 are verifiable entirely on **virtio-blk**, so none of this blocks on
Programme G (AHCI/NVMe).  G is what makes it work on real hardware.

**Compatibility rule for every phase:** `make run` must keep working on the
current single-partition image until F3 lands role-aware mounting, and the
release ISO (ramdisk, volatile) must keep booting throughout — it is the
regression canary that the block contract stayed backend-agnostic.

## 7bis. MAINTAINER ANSWERS (2026-07-23) — binding, with improvements

- **Q1 → full zip support.**  miniz provides both a zip reader and writer, so
  `nxcomp` can create as well as expand archives (needed once the installer
  wants to snapshot a user tree, not only unpack one).  Per-file CRC32 in the
  zip container is checked on extract — cheap, and independent of §4bis.
- **Q2 → tunable with a default** (already stated).  Under D3' it is the read
  cache budget.
- **Q3 → MACHINE holds the kernel partition: immutable EXCEPT authorised
  updates.**  So P1 is not "never writable", it is "writable only through one
  audited path".  **Decision D14 — the update path:** a kernel update is
  (1) staged as a file, (2) verified against its signature/digest BEFORE
  anything is overwritten, (3) written to the INACTIVE of two kernel slots, and
  (4) activated by flipping a pointer in P0.  **Two slots (A/B), not one**: an
  update that fails mid-write must leave a bootable machine, and with a single
  slot there is a window where the only kernel on disk is half-written.  The
  previous slot stays as the rollback target.
- **Q4 → the system must genuinely support MULTIPLE users**; one user was an
  example only.  Nothing in the design hardcodes `usr1`: partitions carry a
  USR role GUID plus a user id, `nxsetup` can create N of them, and the home is
  resolved through `sys.env` per session (the B2.3 decision).  **Decision D15:**
  no code may name `usr1` as a constant — the default installation creates one
  user, which is not the same thing as supporting one.
- **F4 → must not force anything** — accepted and acted on: see D3'/D6'.  The
  verification behind that correction is recorded in §1 (the buffer cache is
  dead code; ext4 is already write-through; the true gap is atomicity).

### Q5 — key, versioning, passwords: the answer, improved

The maintainer's proposal: asymmetric public key in a protected partition;
SHA-256 computed together with the version number; userland declares "supports
up to version N"; root password `root` as a second mechanism.  Three of those
are right; two need changing, and I would not be doing my job by implementing
them as stated.

**Keep — asymmetric, key in a protected partition.**  Correct.  **Decision D16:
Ed25519**, not RSA: 32-byte public key, 64-byte signature, verification is a few
hundred lines with no bignum library, and it is constant-time by construction.
RSA would drag a multiprecision implementation into the kernel for no benefit.

**Keep — bind the version into the signed data.**  Correct and important.

**IMPROVE 1 — a key "in a protected partition" is only as strong as what
protects the partition.**  If the machine level can write P0/P3, then whoever
reaches machine level replaces the key and the whole chain means nothing.  The
honest structure is a **chain where each stage verifies the next and holds that
stage's key**: bootloader → kernel → root.  A chain must terminate in something
software cannot rewrite; on real hardware that is secure-boot/TPM/fuses.
**Decision D17: build the chain now, anchor it later.**  v1: the kernel holds
the public key for ROOT (compiled into the kernel image, itself covered by P1's
Merkle root), P0 holds the root hashes.  Document plainly that until an anchor
exists this defends against corruption and casual tampering, not against an
attacker who already has machine-level write access.  Writing that down is the
difference between security and the appearance of it.

**IMPROVE 2 — "supports up to version N" is the wrong direction for the
dangerous case.**  A ceiling stops the system from loading something too NEW.
The attack is the opposite: feeding it something OLD — a genuinely signed image
from before a fix.  **Decision D18: anti-rollback.**  P0 stores a monotonic
`min_accepted_version`; an image whose signed version is below it is refused
even though its signature is perfectly valid, and a successful update raises the
floor.  Keep the maintainer's ceiling as well (it catches the honest "this
userland is older than this system" mismatch) — but the floor is the one that
is load-bearing.

**IMPROVE 3 — do not ship `root`/`root`.**  A default password that exists in
the source is a permanent credential on every machine ever installed, and it is
the single most reliably exploited weakness in shipped systems.  It is also
unnecessary here: `nxsetup` is already an interactive first-boot program that
asks for a username, so it can ask for the root password in the same step.
**Decision D19:**
- the root password is **set at install**, never compiled in;
- it is stored as a **salted PBKDF2-HMAC-SHA256** hash (buildable on the SHA-256
  this design already adds — no new primitive), never plaintext and never a bare
  digest, so a leaked P0 does not yield the password;
- a **development** build may pre-seed a known password, but only behind an
  explicit build flag that also makes the system announce it at boot, so a dev
  image can never be mistaken for a shipped one.

**As a second factor**, a password is not one — it is the same factor as the
first.  If a genuine second mechanism is wanted later, the natural one here is
possession of the signing key (an update/recovery path authorised by signature
rather than by password), which the chain above already provides.  The
maintainer's "later, a server-based security system" fits exactly there: the
server signs, the machine verifies — no shared secret on the device.

### D20 — the key compartment (maintainer: "an invisible secure partition, kernel-readable only")

Accepted, with one correction that changes WHAT goes in it.

**The correction: a public key does not need secrecy — it needs integrity.**
Hiding the verification key buys nothing (it is public by definition; the
signature verifies whether or not the attacker knows it).  What must be
impossible is *replacing* it: an attacker who swaps the public key can sign
anything.  So the image-verification key belongs where it is WRITE-protected
and covered by the chain — compiled into the kernel image, itself covered by
P1's Merkle root (D17) — **not** in a secret store.  Putting it in a hidden
partition would protect the property it does not need and leave the property it
does need to the hidden partition's write protection, which is weaker.

**What the compartment is genuinely for**, then — the things that are actually
secret:
- the root/user password hashes (salted PBKDF2, D19 — hashed, but still not
  something to leave world-readable);
- **later, the device's own PRIVATE identity key**, when the maintainer's
  server-based scheme arrives: the server signs updates, and the device
  authenticates itself.  That private key is a real secret and this is its home.
  (Today the device holds no private key at all, which is itself a security
  property worth keeping until it must change.)

**Properties, stated precisely:**
- **Not in the VFS namespace.**  Never mounted, no mountpoint, no path — so no
  userland call can name it.  This is the part that is real access control, and
  it is enforced by the same seam as everything else (`vfs_write_allowed` never
  sees it because it is not in the tree at all).
- **Kernel-only reader.**  One reader in the kernel's key-loading path; no block
  device node exposed to userland.
- **"Invisible" = out of the partition table's normal view**, so another OS's
  installer or a disk utility does not offer to format it.  **This is
  anti-accident, not anti-attacker** — a disk imager sees every byte.  Recording
  that distinction is the point: obscurity is a convenience here, not a defence.
- **Confidentiality is bounded by the boot chain.**  Anyone who can boot their
  own kernel, or read the disk offline, reads it.  Real secrecy needs hardware
  (TPM sealing, a secure element, or a key derived from fuses) — the same anchor
  D17 defers.  **Decision: build the compartment now with the interface a
  hardware anchor will need (sealed/unsealed at a single seam), so anchoring is
  a provider swap and not a redesign.**

So the layout gains a fifth region, and P0 splits by property rather than by
convenience: **P0 NEXS-META** (root hashes, install marker, version floor —
integrity-protected, readable) and **PK KEYSTORE** (secrets — confidential,
kernel-only, not in the namespace).
